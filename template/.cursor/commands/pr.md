# PR(Pull Request) 생성

다음 단계를 따라 팀 규칙에 맞는 PR을 생성해주세요:

## 1단계: 브랜치 변경사항 분석
- 현재 브랜치와 base 브랜치(develop 또는 main) 비교
- `git log origin/develop..HEAD` 또는 `git diff origin/develop...HEAD` 확인
- 모든 커밋과 변경된 파일 파악

## 2단계: PR 제목 작성
- 형식: `[타입] 간결한 설명` (50자 이내)
- 타입은 커밋 타입과 동일 (feat, fix, refactor 등)
- 예시: `[feat] 사용자 인증 기능 구현`

## 3단계: PR 본문 작성

다음 템플릿을 사용하여 작성:

```markdown
## 변경 내용
- 주요 변경사항을 bullet point로 나열
- 각 항목은 구체적으로 작성

## 변경 이유
- 왜 이 변경이 필요한지 설명
- 관련 이슈나 배경 설명

## 테스트 방법
1. 로컬 환경에서 테스트한 방법
2. 확인해야 할 주요 케이스
3. 예상되는 결과

## 체크리스트
- [ ] 모든 테스트 통과
- [ ] .cursorrules 규칙 준수
- [ ] 불필요한 console.log 제거
- [ ] 코드 리뷰 준비 완료

## 스크린샷 (선택사항)
UI 변경이 있는 경우 스크린샷 첨부

## 관련 이슈
- Closes #이슈번호
```

## 4단계: 리뷰어 지정 안내
- 최소 1명 이상의 리뷰어 지정 필요
- 관련 도메인 전문가를 리뷰어로 지정 권장

## 5단계: PR 생성 전 체크
- [ ] 브랜치명이 규칙에 맞는지 확인 (feature/*, fix/* 등)
- [ ] 커밋 메시지가 규칙에 맞는지 확인
- [ ] 불필요한 파일 변경이 없는지 확인
- [ ] 민감 정보가 포함되지 않았는지 확인

**주의사항**:
- PR은 작고 명확하게 (300줄 이내 권장)
- 여러 기능이 섞여있다면 PR 분리 고려
- Draft PR로 먼저 올리고 피드백 받는 것도 좋은 방법
