# 커밋 메시지 생성

다음 단계를 따라 팀 규칙에 맞는 **이슈 기반 커밋 메시지**를 생성해주세요:

## 1단계: 변경사항 및 이슈 확인
- `git diff --staged` 명령어로 스테이징된 변경사항 확인
- 변경된 파일과 코드 내용 파악
- **현재 브랜치 이름에서 이슈 번호 추출** (예: `feature/ISSUE-123`)
- 이슈 번호가 없다면 사용자에게 확인 요청

## 2단계: 커밋 타입 결정
다음 중 적절한 타입 선택:
- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `refactor`: 코드 리팩토링 (기능 변경 없음)
- `chore`: 설정/빌드/환경 작업
- `docs`: 문서 수정
- `test`: 테스트 추가 또는 수정
- `style`: 코드 포맷팅, 세미콜론 누락 등

## 3단계: 커밋 메시지 작성 규칙
**형식**: `[작업종류] 상세 내용 (#이슈번호)`

- **제목**: 한글로 작성 (50자 이내)
- **이슈 번호 필수**: (#123) 형태로 제목 끝에 포함
- **본문** (선택사항):
  - 무엇을 변경했는지 (bullet point)
  - 왜 변경했는지

## 4단계: 커밋 메시지 제안
위 규칙에 따라 3가지 옵션을 제안해주세요:
1. **간결한 버전** (제목만)
2. **표준 버전** (제목 + 간단한 본문)
3. **상세 버전** (제목 + 상세 본문)

## 예시
```bash
# 간결한 버전
[feat] 사용자 로그인 API 추가 (#123)

# 표준 버전
[feat] 사용자 로그인 API 추가 (#123)

- JWT 기반 인증 구현
- 로그인 실패 시 에러 처리

# 상세 버전
[feat] 사용자 로그인 API 추가 (#123)

- JWT 기반 인증 구현
- 로그인 실패 시 에러 처리 추가
- bcrypt를 사용한 비밀번호 암호화
- 관련 테스트 코드 작성
```

## GitHub CLI를 사용한 커밋 예시
```bash
# 스테이징
git add .

# 커밋 (제안된 메시지 사용)
git commit -m "[feat] 사용자 로그인 API 추가 (#123)

- JWT 기반 인증 구현
- 로그인 실패 시 에러 처리"

# 푸시
git push origin feature/ISSUE-123
```

**주의사항**:
- **이슈 번호는 필수**입니다. 브랜치명에서 자동으로 추출하거나 사용자에게 확인하세요
- 변경사항이 없으면 커밋하지 마세요
- 여러 기능이 섞여있다면 커밋 분리를 권장합니다
- .cursorrules 파일의 커밋 규칙을 준수하세요
- 한 커밋에는 하나의 논리적 변경사항만 포함하세요
